# Guldly Project Audit — July 10, 2026

Full review of the codebase (10,430 lines across 57 Dart files), the Supabase backend (`njcwivpthvrpqocibrpb`), Stripe integration, and git state. This replaces the earlier project notes, some of which were stale (gold price no longer uses GoldAPI/MetalPriceAPI — it now runs through an edge function backed by Yahoo Finance + Frankfurter).

**Update log:** Items 1–4 below were fixed via Claude Code (prompts 1–3 from `docs/claude-code-prompts.md`) and independently re-verified against the live Supabase project on the same day. See the "Resolved" notes inline. Items 5–12 are still open.

## What's actually done

- Auth: login/register/onboarding, implicit flow (fixes the PKCE email-confirmation bug), password reset/update.
- All core screens built and routed: home, wallet, portfolio, buy (recurring + one-time), sell, gift, delivery, add funds, calculator, profile, security, notifications, more.
- Buy/Sell/Gift/Delivery/Add-Funds are all wired to Supabase via `SECURITY DEFINER` RPCs (`rpc_buy_gold`, `rpc_sell_gold`, `rpc_send_gift`, `rpc_request_delivery`, `rpc_add_funds`) rather than raw table writes — good practice, keeps balance math atomic.
- Home screen and portfolio pull live data from `walletProvider` / `goldPriceProvider`, no hardcoded numbers left.
- Stripe: mobile uses the native Payment Sheet (`card_checkout_sheet.dart`), web uses a Stripe Checkout redirect (`create-payment-intent` edge function, `mode: web_checkout`) since `flutter_stripe` doesn't work on web. Redirect-back flow into `/receipt` is handled with query params.
- Recurring subscriptions: `pg_cron` is installed **and actively scheduled** (`SELECT process_due_subscriptions()` daily at 08:00 UTC, job id 1, active). This works, despite the migration file comment implying it was still manual — it was enabled directly against the project.
- Realtime in-app transaction banners via Supabase Postgres Changes (`notification_service.dart`).
- Gold price edge function (`get-gold-price`) refreshes every 60s client-side and keeps a 20-point history for the mini chart.

## Bugs & gaps, prioritized

### P0 — money & security

**1. ~~"Bank transfer" instantly credits gold/funds with no payment ever collected.~~ — RESOLVED**
`add_funds_screen.dart` and the buy flow both offer a `bank` payment method alongside `wallet` and `card`. Only `card` routes through Stripe. For `bank`, `_onContinue` calls straight into `goldTransactionServiceProvider` → `rpc_buy_gold` / `rpc_add_funds`. Server-side, `rpc_buy_gold` explicitly branches: if `payment_method != 'wallet'` it credits `gold_grams` unconditionally, no balance check, no pending state. So selecting "Banköverföring" credits gold/SEK immediately with zero verification that a bank transfer occurred. The UI even claims "ACH, 3–5 bankdagar" (3-5 business days) but settles instantly. This is either an unfinished feature or a real exploit path — pick one and fix it.

*Fixed (option a): the "Banköverföring" / "Direktöverföring" rows were removed from both `add_funds_screen.dart` and `buy_gold_screen.dart`, and `GoldTransactionService` now throws `UnsupportedError` if `paymentMethod == 'bank'` reaches `buyGoldOnetime`, `addFunds`, or `createRecurringSubscription` — defense-in-depth against any future code path reintroducing it client-side. Verified independently: `rpc_buy_gold` was additionally hardened server-side (see item 2) to reject any non-`wallet` payment method outright, which closes the residual server-side hole that the client-only fix alone wouldn't have. 5 new widget/unit tests added in `bank_transfer_disabled_test.dart`, all passing.*

**2. ~~No server-side verification that a Stripe payment actually succeeded before crediting gold.~~ — RESOLVED**
The client calls `Stripe.instance.presentPaymentSheet()` (or completes Checkout on web), and once that call returns success, the *client* calls `buyGoldOnetime(...)` directly. The RPC has no way to check "was this specific payment actually captured by Stripe" — it trusts whatever `payment_method` string the client sends. A modified client, or a captured/replayed request, could call `rpc_buy_gold` with `payment_method: 'credit_card'` and get gold with no payment at all. This needs either a Stripe webhook that writes the transaction (server is the source of truth) or the RPC verifying the `PaymentIntent` status via a service-role call before crediting.

*Fixed: a new `stripe-webhook` edge function is deployed and active, calling a new `confirm_stripe_payment` Postgres function (service-role only) that credits `wallets`/`transactions` on `payment_intent.succeeded`. Idempotency is enforced with a real `UNIQUE` constraint on `transactions.stripe_payment_intent_id` plus `ON CONFLICT DO NOTHING` — confirmed the constraint exists live and the function returns `'duplicate'` on Stripe's webhook retries rather than double-crediting. The client no longer calls `rpc_buy_gold`/`rpc_add_funds` after a card payment (confirmed by grepping `buy_onetime_screen.dart`, `card_checkout_sheet.dart`, `add_funds_screen.dart` — all now carry comments pointing at the webhook). `rpc_add_funds` and `confirm_stripe_payment` are now `service_role`-only in the database (confirmed via `information_schema.role_routine_grants`) — the client physically cannot call them anymore. `rpc_buy_gold` also now raises an exception server-side for any `payment_method` other than `'wallet'`, so even a direct authenticated REST call can't get free gold out of it.*

**3. ~~Several `SECURITY DEFINER` functions are exposed to `anon`/`authenticated` over PostgREST that shouldn't be.~~ — RESOLVED**
Supabase's advisor flags `handle_gift_sent`, `handle_new_profile`, `handle_new_profile_notifications`, `handle_new_user`, `rls_auto_enable`, and `process_due_subscriptions` as callable directly via `/rest/v1/rpc/...` by any signed-in (some even anonymous) user — these are meant to run only as triggers/cron, not as public endpoints. `process_due_subscriptions` in particular being anon-callable means anyone could trigger recurring billing runs on demand. Revoke `EXECUTE` from `anon`/`authenticated` on all of these.

*Fixed via migration `20260710124602_security_hardening`. Confirmed live: `EXECUTE` on `handle_gift_sent`, `handle_new_profile`, `handle_new_profile_notifications`, `handle_new_user`, `rls_auto_enable`, and `process_due_subscriptions` is revoked from both `anon` and `authenticated` — none show up in `role_routine_grants` for those roles anymore. The user-facing `rpc_buy_gold`/`rpc_sell_gold`/`rpc_send_gift`/`rpc_request_delivery` remain correctly `authenticated`-only (not `anon`), and `rpc_add_funds` was intentionally *not* re-granted to `authenticated` since it's now webhook-only (see item 2).*

**4. ~~Mutable `search_path` on 9 functions.~~ — RESOLVED**
`handle_new_profile`, `handle_new_profile_notifications`, `rpc_sell_gold`, `rpc_send_gift`, `rpc_request_delivery`, `handle_gift_sent`, `process_due_subscriptions`, `rpc_buy_gold`, `rpc_add_funds` — none pin `search_path`. Standard Postgres hardening: add `SET search_path = public, pg_temp` to each definition.

*Fixed in the same `security_hardening` migration. Re-ran the security advisor: the `function_search_path_mutable` warning is gone entirely — 0 remaining instances across all functions, including the new `confirm_stripe_payment`.*

**5. Leaked password protection is disabled in Supabase Auth.** Free to turn on (checks against HaveIBeenPwned), currently off. Still open — this is an Auth dashboard/API setting, not something a migration can flip; it's the only item left in the security advisor's report as of this update.

### P1 — functional gaps

**6. ~~Gifting to someone without a Guldly account silently destroys the gold.~~ — RESOLVED**
`rpc_send_gift` debits the sender unconditionally. The recipient side only happens in the `handle_gift_sent` trigger, which looks up `auth.users` by email and does nothing if no match is found — no error, no pending/claimable gift, no refund. The sender's gold is just gone. There's no UI warning about this either.

*Fixed via migration `20260710134252_pending_gifts`: a new `pending_gifts` table (keyed by lowercased recipient email, RLS-readable only by the sender, nothing client-writable) captures unmatched gifts instead of dropping them. `handle_gift_sent` now does case-insensitive email matching (the old version was case-sensitive — a second silent-loss path that wasn't in the original writeup) and flips the sender's transaction to `pending` instead of `completed` when there's no match yet. `handle_new_user` claims any pending gifts at signup, crediting the new wallet and settling the sender's transaction. `rpc_send_gift` now returns `'delivered'` / `'pending'` so `gift_screen.dart` → `receipt_screen.dart` can show "Gåvan väntar på mottagaren" with an explanation instead of implying instant delivery. Verified live: sent a real 0.25 g gift to a non-existent email, confirmed sender debited + pending row created + sender tx `pending`; then created the matching `auth.users` row and confirmed recipient credited, `gift_received` tx written, pending row claimed, sender tx settled to `completed` — all reverted afterward, script saved at `docs/pending-gifts-verification.sql`.*

*Found in the process, fixed same session: while testing this, a forged-insert attack against `transactions` succeeded and minted 9999 g into the real `ebooood94@gmail.com` wallet. Root cause: `transactions` had two overlapping permissive INSERT policies (`"transactions: own insert"` from migration 001, and a never-dropped original `"Users can insert own transactions"`) — Postgres OR's permissive policies, so a client could insert `transactions` rows directly (bypassing the RPCs entirely) as long as `user_id = auth.uid()`, with no server-side validation of amounts. Closed via `20260710134750_drop_duplicate_transactions_insert_policy` — `transactions` now has zero INSERT policies, so only the `SECURITY DEFINER` RPCs (which bypass RLS) can write rows. Verified live: the wallet is back to its exact pre-test balance (1.0407 g / 10 362.71 kr) and `transactions` has no INSERT policy at all.*

*Cleaned up via migration `20260710_dedupe_rls_policies` (007): dropped the 7 remaining legacy `"Users can …"` policies across `profiles`, `wallets`, `subscriptions`, `notification_preferences`, and (found while in there, same pattern) a leftover duplicate `SELECT` on `transactions`. Two of the dropped policies (`subscriptions`, `notification_preferences`) were `FOR ALL`, which implicitly granted client-side `DELETE` that the newer per-action policy sets don't — confirmed via repo-wide grep that `lib/` contains zero `.delete()` calls against Supabase, so nothing relied on that. Verified live: every `(table, action)` pair under `public` now has exactly one policy — `profiles` (read/insert/update), `wallets` (read/insert/update), `subscriptions` (read/insert/update), `notification_preferences` (all), `transactions` (read), `pending_gifts` (read), `push_tokens` (all) — no legacy names remain anywhere.*

**7. ~~Push notifications don't exist end-to-end.~~ — RESOLVED (option b: honest UI, not real delivery)**
`NotificationsScreen` lets users toggle `push_price_alerts`, `push_transaction_updates`, etc., and they persist to `notification_preferences`. But `NotificationService.savePushToken()` — the only code that would register a device for push — is never called anywhere in the app (no FCM/APNs setup, no permission request, no token registration on login). The `push_tokens` table has 0 rows. These toggles currently do nothing beyond storing a preference. Either wire up real push (Firebase Cloud Messaging is the natural fit given no other push infra exists) or relabel these as "email preferences only" until it's built.

*Went with (b) rather than (a): standing up FCM (Firebase project, platform credentials, web-push VAPID/service worker, per-platform permission flows) from zero is a real infrastructure build, and this app's actual shipping surface — web + Windows — is exactly where FCM is weakest. Building it now, mid payments/security hardening, wasn't the right sequencing call. Correctly scoped wider than the original ask, too: email and SMS were making the same false "you'll be notified" promise as push despite having zero send path either, so `notifications_screen.dart` now shows a `BannerVariant.warning` `InfoBanner` ("Aviseringar levereras inte ännu – varken push, e-post eller SMS...") and all three section headers are suffixed "(KOMMER SNART)", not just the push one. Toggles still persist to `notification_preferences` unchanged, so no data loss whenever (a) does get built. Verified independently: no `firebase` dependency anywhere in `pubspec.yaml`, `NotificationService.savePushToken`/`.initialize` are still uncalled outside their own file, `BannerVariant.warning` already existed in `InfoBanner` (reused, not reinvented), and `flutter analyze`/`flutter test` are unaffected (still 5/5 passing — this was a pure copy change, nothing new to test).*

*Follow-up plan for option (a), whenever push becomes a priority: add `firebase_messaging`, wire `savePushToken()` into first-login (the `push_tokens` table + idempotent upsert are already there waiting), and add a Supabase edge function that calls FCM's HTTP v1 API on relevant `transactions` inserts.*

**8. ~~Duplicate signup triggers.~~ — RESOLVED**
`handle_new_profile` + `handle_new_profile_notifications` (older, minimal) appear to do the same job as the newer `handle_new_user` (which inserts profile + wallet + notification_preferences with `ON CONFLICT DO NOTHING`). If both trigger sets are still attached to `auth.users`, signup does redundant work — harmless today because of the conflict guards, but it's a landmine for whoever touches this next. Consolidate to one.

*Turned out sharper than originally described: the redundancy wasn't two triggers on `auth.users` — it was `on_auth_user_created` → `handle_new_user` on `auth.users`, cascading into two **unguarded** triggers on `public.profiles` (`handle_new_profile`, `handle_new_profile_notifications`) that inserted the wallet and notification-preferences rows with no `ON CONFLICT`. Signup only ever produced one row each because the unguarded cascade happened to run first and `handle_new_user`'s guarded inserts then no-op'd — but any direct `profiles` insert racing against that cascade (e.g. the app's `firstLoginSetupProvider`, which upserts `profiles` and `wallets` concurrently via `Future.wait`) could have hit a unique-constraint violation. Fixed via migration `20260710130839_single_signup_trigger`: both `profiles` triggers and their functions dropped, `handle_new_user` also had its `search_path` aligned to `public, pg_temp` (it was missing the `pg_temp` half). Verified independently: live trigger inventory across `public`/`auth` is now exactly `on_auth_user_created → handle_new_user` and `on_gift_sent → handle_gift_sent`, and `handle_new_profile`/`handle_new_profile_notifications` no longer exist as functions at all (not just detached).*

### P2 — consistency & polish

**9. ~~Hardcoded colors bypass the design system.~~ — RESOLVED**
`splash_screen.dart` and `portfolio_screen.dart` use raw `Color(0x...)`/`Colors.*` instead of `AppConstants` tokens, breaking the project's own styling rule (CLAUDE.md: "Never hardcode a `Color(0x...)` or `Colors.*` in a screen").

*Fixed: all eight literals across the two files replaced with `AppConstants` tokens — `splash_screen.dart` (2× `Colors.transparent` status-bar overlay, `Color(0xFF0F0F0E)` splash bg, `Colors.white` button label) and `portfolio_screen.dart` (`Color(0xFFCDE8DA)` return-tile icon bg, `Colors.transparent` sheet barrier, 3× `Colors.white` for sheet surface / button fg / spinner). Four new semantic tokens added since none mapped to an existing one: `onGold` (white on gold CTAs), `splashBackground` (`0xFF0F0F0E`), `transparent` (`0x00000000`), `returnIconBg` (`0xFFCDE8DA` green tint); the edit-sheet's white surface reused the existing `card` token rather than adding a duplicate. Purely a token-indirection swap — every new token holds the exact ARGB value it replaced, so no visual change. `flutter analyze` clean; both files now have zero color literals (confirmed by grep).*

**10. ~~Zero automated test coverage.~~ — RESOLVED**
`test/widget_test.dart` is an empty stub (`void main() {}`). No widget tests, no unit tests on the gold-price math, wallet math, or the RPC-calling providers.

*Fixed: suite went from 0 to 25 passing tests (`flutter analyze` clean). `models_test.dart` hand-verifies `GoldPrice.pricePerGramSek` against an independently chosen input (`pricePerOzUsd=3110.35, usdToSek=10.0` → exactly `1000.0`, re-derived and confirmed: `3110.35 × 10 / 31.1035 = 1000.0`) plus an arbitrary case, a zero-input guard, and `fromMap`/round-trip coverage for `Wallet`, `UserProfile`, `Transaction`, `Subscription`, and `NotificationPreferences`. `payment_method_mapping_test.dart` covers the `card`/`bank`/`wallet`/unknown → db-string mapping. `buy_onetime_screen_test.dart` and `add_funds_screen_test.dart` verify pill-tap-sets-amount and continue-button enable/disable, with all backend providers swapped via `ProviderScope` overrides so nothing hits the network. One small production change to enable testing: `GoldTransactionService._paymentToDb` was made `@visibleForTesting` and public as `paymentToDb` — no behavior change, both internal call sites updated. Test counts verified independently per-file (13+4+2+1+5 = 25, matching).*

*Two things surfaced while writing the tests, both still open, both low-priority: (1) `lib/core/models/gold_price.dart` defines a second, unused `GoldPrice` class (with a 2% buy/sell spread) — confirmed via grep that nothing in `lib/` imports it; the live model is in `models.dart`. Dead code, worth deleting. (2) The amount-suggestion pills hard-code an ASCII space in labels like `'5 000 kr'`, while the amount hero formats via `NumberFormat('sv_SE')`, which uses a non-breaking space for grouping — so the two don't match character-for-character on grouped amounts. Purely cosmetic today (nothing currently asserts equality between them outside the new tests, which sidestepped it), but worth knowing before writing future tests or copy that compares them.*
